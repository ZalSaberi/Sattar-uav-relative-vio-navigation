# Phase 3 MSCKF/VIO Math and Consistency Review

## Scope

Phase 3 reviews MSCKF/VIO math and consistency risks after the Phase 0-2 runtime baseline. No source code changes were made in this review. The goal is to identify fixes that should be approved before implementation.

- Base branch: `codex/runtime-baseline`
- Working branch: `codex/phase3-msckf-review`
- Starting commit: `2ae28d7 docs: add phase 2 validation report`

## Review Findings

| Area | Classification | Evidence | Impact | Recommended action |
|---|---|---|---|---|
| Chi-square gating confidence level | definite bug | `src/msckf.py` builds the table with `chi2.ppf(0.05, i)` while the comment says confidence level 0.95, and `gating_test()` accepts `gamma < threshold`. The code also passes `len(cam_state_ids)-1` or `len(involved_cam_state_ids)` as DOF, while the nullspace-projected residual has `4*n - 3` rows for stereo observations. | The 5 percent quantile rejects most valid measurements under the assumed model. Wrong DOF further distorts gating and can reject or accept updates unpredictably. | Use 0.95 confidence unless a different confidence is intentionally documented. Use the actual residual dimension after nullspace projection, usually `len(r_j)`, and extend/check the table range. |
| Covariance update form | likely bug | `measurement_update()` computes `state_cov = (I - K H) P` and comments out the Joseph form. It then symmetrizes the result. | The simplified form can lose positive semi-definiteness in finite precision, especially with large updates and repeated QR-reduced updates. Symmetrization hides asymmetry but does not guarantee PSD. | Prefer Joseph form, `P = (I-KH)P(I-KH).T + K R K.T`, or prove and test an equivalent numerically stable update. Add symmetry and eigenvalue sanity checks. |
| Feature initialization and depth validation | likely bug | `generate_initial_guess()` divides by `a @ a` without a degeneracy guard. `initialize_position()` assumes valid `cam_poses[0]` and `cam_poses[1]`, solves `np.linalg.solve(...)` without catching singular matrices, and accepts the result based only on positive depth. `translation_threshold` is currently disabled with `-1.0`. | Degenerate parallax, poor stereo geometry, or singular normal equations can produce NaN/inf or bad feature positions that feed the filter. Disabled motion checks make this more likely. | Add finite checks, denominator guards, singular-solve handling, reprojection-cost thresholding, and a real parallax/motion threshold before accepting initialized features. |
| Measurement Jacobians | likely bug | `measurement_jacobian()` computes `H_f` from projection derivatives, then overwrites it with `H_f = -H_x[:4, 3:6]` after modifying `H_x` with an observability-style projection. The derivation is not documented or tested. | If `H_f` is inconsistent with the residual and camera Jacobian, nullspace projection removes the wrong feature subspace and corrupts the MSCKF update. | Derive the Jacobian convention explicitly and compare analytic Jacobians against finite differences for cam0/cam1 stereo residuals before changing equations. |
| Positive depth checks | likely bug | Feature initialization checks final positive depth across initialization poses, but projection code divides by `p_c0[2]` and `p_c1[2]` in `measurement_jacobian()` with no near-zero, finite, or behind-camera guard. | A feature that later becomes near-zero depth or behind a camera can produce unstable residuals and update spikes. | Reject non-finite features and measurements with non-positive or very small depth before building Jacobians. Keep the final initialization positive-depth check. |
| IMU initialization assumptions | design/scientific concern | `initialize_gravity_and_bias()` averages the first 200 buffered IMU samples after the selected dataset start/offset, assumes they are stationary, sets gyro bias from mean gyro, estimates gravity direction/magnitude from mean acceleration, and leaves accelerometer bias at zero. | This can be reasonable for stationary starts, but offset runs may begin while moving. Gravity and bias initialization can then be biased and affect all later propagation. | Document the stationary-start assumption, optionally initialize only from known stationary windows, and compare with EuRoC ground truth/IMU bias fields where available. |
| Camera-state pruning logic | design/scientific concern | `find_redundant_cam_states()` uses hard-coded thresholds `0.2618`, `0.4`, and `0.5` instead of config values, removes exactly two states, and reuses the same gating DOF issue during pruning. The Phase 2 arccos clipping guard is okay for now. | The pruning policy may be acceptable as a starting heuristic, but it is not clearly tied to the configured thresholds or to the measurement model. Wrong pruning-gate DOF can drop useful constraints. | Move thresholds to config or use existing config values. Fix gating DOF first, then review pruning behavior against a reference MSCKF policy. |
| IMU/image callback synchronization and shared mutable state | definite bug | `VIO` runs image, IMU, and feature workers in separate threads. `IMUProcessor.imu_buffer` is appended by the IMU thread and read/sliced by the image thread without locks. `MSCKF.imu_msg_buffer` is appended by the IMU thread and read/sliced by the feature thread without locks. There is no timestamp barrier ensuring all IMU samples up to an image timestamp have arrived before processing that feature message. | Race conditions and timing skew can make feature tracking and state propagation depend on thread scheduling rather than dataset timestamps. This can also leave the state timestamp behind the image timestamp. | Add synchronization around shared buffers or move to a single ordered event loop. Ensure feature processing waits for IMU coverage through the image timestamp, with interpolation if needed. |
| Output format and trajectory evaluation readiness | okay for now, with concerns | Output is TUM-like: `timestamp p_x p_y p_z q_x q_y q_z q_w`, one file per dataset/offset, and repeated runs overwrite unless append is requested. However `_write_state()` uses `imu_state.timestamp`, while `publish(time)` receives the feature timestamp separately. | The text format is usable for simple plotting, but evaluation can be timestamp-shifted if propagation stops at the last IMU sample before the image timestamp. Accuracy still needs alignment and comparison to EuRoC ground truth. | Keep the format for now, but verify timestamp semantics before ATE/RPE. Add an evaluation script that aligns estimated IMU/body/camera pose with EuRoC ground truth and reports ATE/RPE. |

## Priority Table for Phase 3 Fixes

| Priority | Fix | Classification | Rationale |
|---|---|---|---|
| P0 | Correct chi-square confidence and gating DOF. | definite bug | Directly controls whether feature updates enter the filter. Current 0.05 threshold conflicts with the documented 0.95 confidence level. |
| P0 | Remove threading races or enforce timestamp-ordered processing. | definite bug | Filter behavior should depend on dataset timestamps, not worker scheduling. Shared mutable IMU buffers are accessed without synchronization. |
| P0 | Add finite, denominator, and positive-depth guards before feature initialization and Jacobian construction. | likely bug | Prevents NaN/inf residuals and unstable updates from bad geometry. |
| P1 | Review and replace covariance update with Joseph form or a proven stable equivalent. | likely bug | Needed for covariance PSD and long-run consistency. |
| P1 | Validate measurement Jacobians against finite differences and document frame conventions. | likely bug | This is central MSCKF math and should be fixed only with derivation/test evidence. |
| P1 | Re-enable or redesign parallax/motion checks for feature initialization. | design/scientific concern | Reduces bad depth estimates before they enter the filter. |
| P2 | Make camera-state pruning thresholds configurable and retest pruning with corrected gating. | design/scientific concern | Lower immediate risk than gating/Jacobians, but affects constraint retention. |
| P2 | Clarify IMU initialization assumptions and add stationary-window validation. | design/scientific concern | Important for offset runs and datasets that do not start stationary. |
| P2 | Add trajectory evaluation tooling for EuRoC ground truth, ATE, and RPE. | okay for now, with concerns | Output is usable, but accuracy claims need quantitative comparison. |

## Summary

The highest-confidence Phase 3 bugs are the chi-square gate construction/DOF mismatch and the unsynchronized multi-threaded IMU/image processing path. The most important math risks after that are feature-depth degeneracy handling, covariance update consistency, and measurement Jacobian validation. The output format is usable for smoke runs, but it is not yet enough for scientific evaluation without timestamp validation and EuRoC ground-truth alignment.
